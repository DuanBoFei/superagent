# 01-产品形态调研

## 1. 文档信息

| 项目 | 内容 |
|------|------|
| **调研目标** | 为"生产级通用桌面 AI Agent"项目提供产品形态决策依据，全面梳理同类产品（代码 Agent / 通用 Agent / 桌面 Computer Use / 国内产品）的功能边界、信息架构、交互模式、AI 化程度和形态演变路径 |
| **调研日期** | 2026-06-12 |
| **信源覆盖度** | 主要信源为 Claude 内置 WebSearch（偏海外/GitHub/英文）。**注意**：muyu-search-mcp 工具在当前环境中不可用，国内产品信息同样通过 WebSearch 获取（百度开发者社区、阿里云开发者社区、腾讯云、InfoQ 中文站等）。所有标注 `[WebSearch]` 的事实均指通过 WebSearch 工具获取，部分中文来源另有说明 |
| **涉及产品数量** | 共计 20 个产品（代码 Agent 8 个 + 通用 Agent 5 个 + 桌面/Computer Use 3 个 + 国内产品 4 个） |
| **版本时效** | 所有信息截至 2026 年 5-6 月公开可获取的最新版本 |

### 5.2 关键发现

1. **代码 Agent 正"溢出"到通用场景**：Claude Code 用户已将其用于非编码任务（假期研究、邮件整理）。Copilot CLI 2026 年也开始支持非 PR Agent 任务（Issue 摘要、报告生成）。代码 Agent 的 CLI + 工具调用基础设施天然适合泛化为通用 Agent [WebSearch]

2. **通用 Agent 正"下沉"到代码场景**：OpenClaw 和 Hermes 内建了 Shell 执行、文件操作等代码能力，Manus 有 CodeAct 引擎。纯"通用"和纯"代码"的边界正在模糊 [WebSearch]

3. **Computer Use 当前实用性有限**：Anthropic CU 每个动作 3-5 秒延迟使其不适合时间敏感任务；Google Mariner 因性能和误差率问题于 2026 年 5 月关闭。但它的"万能接口"理念（任何 UI 替代 API 集成）有长期价值 [WebSearch]

4. **多 Agent 是 2026 年的决定性趋势**：Claude Code、Cursor、Copilot、Devin、OpenClaw 均已实现多 Agent 并行。Devin 的上下文隔离发现（Review Agent 看到 10 倍更多问题）是一个关键架构洞见 [WebSearch]

5. **长期记忆是未解决的前沿**：大多数产品的"记忆"仍然基于文件系统（Markdown）+ 向量数据库的组合。Hermes 的自进化闭环（Nudge + Skill 自动生成）和 OpenClaw 的 MemTier 插件（PPO 框架优化检索）代表了前沿探索 [WebSearch]

## 6. CLI → Desktop 形态演变路径分析

### 6.1 行业演变三阶段

| 阶段 | 时间 | 特征 | 代表事件 |
|------|------|------|---------|
| **阶段 1：行内补全** | 2021-2022 | 单行代码建议，被动响应 | GitHub Copilot 发布（2021） |
| **阶段 2：Chat + 项目感知** | 2023-2024 | 对话式交互，仓库级上下文理解 | Cursor Chat、Copilot Chat 普及 |
| **阶段 3：Agent 自主执行** | 2025-2026 | 规划→编辑→测试→迭代→交付全自主 | Claude Code、Cursor Agent、Copilot Agent Mode [WebSearch] |

### 6.2 各产品 CLI→Desktop 演变路径

#### Claude Code：CLI 先行，逐步补全 Desktop
```
CLI (2025 初) → VS Code/JetBrains 扩展 → Slack 集成 → Desktop App (2026.4 重新设计) → Web IDE (claude.ai/code)
```
**策略**：先通过 CLI 占领开发者心智，再用 IDE 扩展满足 GUI 偏好用户，Desktop App 用于非开发者场景（Claude Cowork）[WebSearch]

#### GitHub Copilot：IDE 先行，向下补全 CLI
```
VS Code 扩展 (2021) → 全 IDE 覆盖 → Chat (2023) → Agent Mode (2025) → Copilot CLI (2026.2 GA) → Desktop App (2026.5 TP)
```
**策略**：利用 IDE 入口的庞大用户基础，逐步增加 Agent 自主性，CLI 补全终端工作流 [WebSearch]

#### Cursor：IDE 先行，CLI + Cloud 扩展
```
VS Code fork IDE (2023) → Composer (2024) → 多 Agent 并行 (2025.10) → CLI (2026.1) → Cloud Agents + BugBot
```
**策略**：从最好的 AI IDE 体验出发，CLI 覆盖 CI/CD 和 headless 场景，Cloud 覆盖异步/后台任务 [WebSearch]

#### Devin：Cloud Web 先行，IDE + CLI 后发
```
Cloud Web Platform (2024.3) → Slack/Jira 集成 → Windsurf IDE (2024) → Devin Desktop + CLI (2026.6 更名)
```
**策略**：Cloud VM 全自主 → 用 IDE 级工具（Windsurf）覆盖本地开发者 → 品牌统一为 Devin [WebSearch]

#### Codex CLI：CLI 先行，多客户端统一
```
CLI (2025.4) → VS Code/Web/Desktop/JetBrains/Xcode (2025-2026) → App Server 协议统一所有客户端 (2026.2)
```
**策略**：先以开源 CLI 建立社区和信任，再用 JSON-RPC App Server 协议统一所有前端界面 [WebSearch]

### 6.3 演变规律总结

1. **CLI 和 IDE 的对立已瓦解**：到 2026 年，所有主要产品都同时提供 CLI 和 IDE 入口。"终端 vs IDE"不再是非此即彼的选择 [WebSearch]

2. **CLI 的战略回归**：AI Agent 首选 Shell 二进制文件而非 SDK，因为 `curl` 和 CLI 是从自然语言任务到工作集成的最短路径。这导致了 CLI 的复兴 —— Anthropic（Claude Code）、OpenAI（Codex CLI）、Google DeepMind（Gemini CLI）全部发布了终端工具 [WebSearch]

3. **中间状态：消息平台集成**：Devin（Slack/Teams）、OpenClaw（20+ 平台）、Claude Code（Slack）表明消息平台成为 Agent 的重要"非 IDE 非 CLI"入口。适合异步任务委派 [WebSearch]

4. **Desktop App 是最后拼图**：
   - Claude Cowork（2026.1）专为非开发者设计，目标是桌面自动化
   - Copilot Desktop App（2026.5 TP）目标是"独立的 Agent 驱动开发体验"
   - Devin Desktop（2026.6）定位为"带 Agent 管理器的完整 IDE"
   - Desktop App 的共同特点：并行会话 + Git worktree 隔离 + 可视化 Agent 管理 [WebSearch]

5. **云端 Agent 是第三维度**：Cursor Cloud Agents、Devin Cloud、Codex Cloud、Copilot Cloud Agents 构成了"云端异步执行"的第三个维度。开发者委派任务后关闭设备，Agent 在云端持续工作 [WebSearch]

## 7. 关键发现与启示

### 7.1 对"生产级通用桌面 AI Agent"项目的启示

1. **入口策略：CLI 优先 + 消息平台 + Desktop App 三阶段**
   - 第一阶段：CLI（技术用户、早期采用者）—— 与 Claude Code / Codex CLI 现有生态兼容
   - 第二阶段：消息平台集成（微信/钉钉优先，中国本地化关键差异）—— Hermes/OpenClaw 证明了 20+ 平台的可行性
   - 第三阶段：Desktop App —— 用于 Computer Use 和可视化 Agent 管理

2. **信息架构建议：采用"配置分层 + Markdown Rules + MCP + Hooks"组合**
   - 分层配置（全局 → 项目 → 本地）是行业标准，必须支持
   - `CLAUDE.md` / `AGENTS.md` 格式的 Markdown Rules 文件是 Agent 指令的事实标准
   - MCP 协议已成为跨产品通用协议，必须支持
   - Hooks 系统是区分"工具"和"平台"的关键特征，应作为核心功能而非后期附加

3. **交互模式建议：Plan 显式展示 + 实时流式 + 多干预方式**
   - 参考 Claude Code 的"先 Plan 等批准再执行"范式（对大型任务）
   - 同时提供 Copilot CLI 风格的"Autopilot 模式"（对信任的日常任务）
   - 干预方式至少提供：批准/拒绝单步操作、逐文件 Diff 审查、Git-原生回滚

4. **AI 能力优先级：代码 > Browser Use > Computer Use**
   - 代码能力是 MVP 核心（所有竞品验证了需求刚性）
   - Browser Use 是第二优先级（网页操作是通用 Agent 最频繁的场景）
   - Computer Use 当前实用性和性能有限，可作为差异化亮点，但不宜作为 MVP 核心

5. **多 Agent 架构：应作为 Day 1 设计考量**
   - Devin 的上下文隔离发现（Reviewer 比 Writer "聪明 10 倍"）是重要架构洞见
   - 建议：主 Agent（编排 + 决策）+ 专用 Sub-agent（Explore / Code / Review）+ 隔离上下文模式

6. **长期记忆：先做简单再做复杂**
   - 起步：Markdown 文件 + 向量数据库（行业标准，够用）
   - 进阶：参考 Hermes 的自进化 Skill 生成（但需注意 skill pollution 风险）
   - 差异化：中文项目的 RepoWiki 式知识图谱（参考通义灵码）

7. **中国市场的独特机会**
   - 国内产品的多模型策略（Qwen + DeepSeek + GLM + Kimi）比海外（Claude-only 或 GPT-only）更灵活
   - 微信/钉钉/飞书的深度集成是海外产品无法提供的差异化
   - 企业 VPC 私有化部署是国内中大型企业的刚需（所有国内头部产品均支持）
   - 完全免费策略（Fitten Code）在国内仍存在，但可持续性存疑
